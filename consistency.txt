# Video Production Backend API – Architecture Specification

## Overview

This project is a backend API for an AI-assisted educational video production system.

The backend is implemented in **Python** using **FastAPI** and is designed to orchestrate the complete production pipeline of a video series. The backend is intentionally designed to be independent of the frontend so that it can later support multiple clients, including:

* Next.js web application
* Command Line Interface (CLI)
* Desktop applications
* Mobile applications
* Automation scripts
* External integrations

The backend exposes REST APIs that allow clients to create projects, execute production stages, manage generated assets, and retrieve production status.

---

# Primary Design Goals

The architecture follows these principles:

* Clean Architecture
* Modular Services
* Separation of Concerns
* AI Provider Independence
* File-Based Asset Management
* Future Database Compatibility
* Future Workflow Automation
* Easy Integration with Multiple LLM Providers

The project is intentionally designed so that every production stage can execute independently.

# Layer Responsibilities

## API Layer

Responsible only for:

* HTTP endpoints
* Request validation
* Response formatting
* Authentication (future)

The API layer must never contain production logic.

---

## Services Layer

Responsible for business logic.

Examples:

* Create project
* Generate narrative
* Generate production script
* Generate narration
* Generate images
* Generate video clips
* Post production

Every service should be independently executable.

---

# Workflow Philosophy

Workflow orchestration is intentionally manual in the initial versions.

Each production stage can be executed independently.

Later versions will include:

* Automatic pipeline execution
* Job queues
* Background workers
* Workflow engine

Current priority is understanding and validating every production stage.

---

# AI Provider Philosophy

The application must never become dependent on a single AI provider.

Every provider must implement common interfaces.

Example

LLMProvider

generate()

chat()

stream()

ImageProvider

generate()

edit()

upscale()

AudioProvider

generate()

clone_voice()

VideoProvider

generate()

extend()

Future providers can be added without modifying business services.

---

# Coding Standards

Use:

* Type hints
* Pydantic models
* pathlib instead of os.path
* Dependency Injection where appropriate
* Logging instead of print()
* Async endpoints when beneficial
* Small focused classes

Avoid:

* Business logic inside routes
* Hardcoded paths
* Hardcoded API keys
* Duplicate code
* Provider-specific logic inside services

# Long-Term Vision

This backend is intended to become a reusable AI production engine rather than a project-specific API.

It should support multiple educational video series, multiple AI providers, multiple frontend applications, and evolving production workflows without requiring architectural changes. Every new feature should integrate into the existing layered architecture by extending services, providers, or storage components rather than bypassing them.
